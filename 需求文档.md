# 番茄钟软件需求文档

## 1. 项目概述

本项目目标是开发一个面向 Windows 平台的轻量级番茄钟软件。产品强调三件事：

1. 计时过程常驻且低打扰
2. 到点后能以足够强的方式提醒用户休息
3. 整体视觉简洁、美观、轻量，兼顾 macOS 风格质感与 Windows 可用性

当前阶段建议先完成 Windows 单平台 MVP，后续再考虑 Linux 适配。

## 2. 原始需求整理

根据 `需求.md` 与 `需求分析.md`，核心需求如下：

1. 每 25 分钟触发一次全屏休息提醒
2. 平时以极小形态存在，可收起或保持为小悬浮窗
3. 悬浮窗可显示当前倒计时，可拖动，不明显占用屏幕空间
4. 点击悬浮窗后可打开主界面
5. UI 要美观，有 macOS 风格参考感
6. 软件需要启动快、占内存小、运行流畅
7. 当前仅要求支持 Windows，后续再支持 Linux

## 3. 产品定位

这不是一个功能很多的任务管理软件，而是一个“高体验要求的桌面常驻工具”。

产品核心由两部分构成：

1. 常驻的轻量倒计时悬浮窗
2. 到时触发的全屏休息提醒界面

因此，设计重点不是复杂业务，而是：

1. 常驻时是否足够轻、足够不打扰
2. 提醒时是否足够明显但不过分冒犯
3. 交互是否顺滑、视觉是否精致

## 4. 目标用户与使用场景

### 4.1 目标用户

适合以下用户：

1. 长时间坐在电脑前办公、学习、编程的人
2. 希望被规律提醒休息，但不想被复杂功能干扰的人
3. 对界面质感、性能和桌面常驻体验有要求的人

### 4.2 典型使用场景

1. 用户开机后软件自动启动，以小悬浮窗常驻桌面
2. 用户开始一个 25 分钟专注周期，悬浮窗显示剩余时间
3. 倒计时结束后，弹出全屏休息页提醒用户离开屏幕休息
4. 用户可以等待休息倒计时结束，也可以跳过或延后提醒
5. 关闭主界面后，软件继续在托盘和悬浮窗中运行

## 5. 功能需求

### 5.1 计时器机制

1. 默认专注时长为 25 分钟
2. 默认短休息时长为 4 分钟
3. 每完成 4 个番茄周期后，进入一次长休息，时长为 8 分钟
4. 计时器支持开始、暂停、重置
5. 休息结束后，不自动进入下一轮专注，由用户手动点击开始

### 5.2 悬浮窗

1. 支持小尺寸常驻显示
2. 可拖动位置
3. 默认始终置顶
4. 显示当前倒计时
5. 显示当前状态，例如“专注中”“休息中”
6. 点击后打开主界面
7. 体积要小，但要保证可点击、可拖动，不影响命中率

### 5.3 主界面

主界面负责集中操作和配置：

1. 开始 / 暂停 / 重置计时器
2. 设置专注时长、短休息时长、长休息时长
3. 查看当前处于第几个番茄周期
4. 设置是否开机自启
5. 设置是否悬浮窗置顶
6. 设置提醒行为

当前不需要：

1. 任务管理
2. 今日统计 / 历史统计
3. 白噪音
4. 键盘快捷键

### 5.4 全屏提醒

1. 专注结束后弹出全屏休息提醒
2. 休息提醒需覆盖所有显示器
3. 全屏页面显示：
   - 当前提示文案
   - 休息倒计时
   - 跳过按钮
   - 延后 5 分钟按钮
   - 关闭提醒按钮
4. 提醒方式需要明显，但不建议设计成完全不可关闭的强制拦截

### 5.5 托盘与后台运行

1. 关闭主窗口后最小化到系统托盘
2. 托盘菜单至少包含：
   - 打开主界面
   - 开始 / 暂停
   - 退出程序
3. 支持开机自启
4. 配置需要持久化保存

## 6. 非功能需求

### 6.1 性能

1. 软件启动要快
2. 常驻内存占用要低
3. 长时间运行稳定，不明显漂移、不明显卡顿

### 6.2 易用性

1. 悬浮窗不遮挡主要工作区域
2. 全屏提醒足够清晰，但不造成过强打断
3. 主界面操作路径要短，基础操作不超过 1 到 2 次点击

### 6.3 视觉风格

1. 可以借鉴 macOS 的简洁、圆角、半透明、轻拟物质感
2. 不能机械照搬 macOS 组件样式，否则在 Windows 上会显得违和
3. 视觉目标应是“精致、轻、安静”，而不是“花哨”

## 7. 当前已经明确的产品决策

根据 `需求分析.md`，当前可以认为已明确的点有：

1. 短休息 4 分钟
2. 每 4 个番茄后长休息 8 分钟
3. 用户允许跳过提醒、延后 5 分钟、关闭提醒
4. 悬浮窗始终置顶，但面积要小
5. 不需要任务统计
6. 主窗口关闭后最小化到托盘
7. 需要开机自启
8. 全屏提醒需要覆盖多显示器
9. 不需要声音、白噪音、快捷键

## 8. 仍需补充确认的问题

虽然整体需求已经足够启动 MVP，但以下细节最好在开发前定稿：

1. 全屏提醒默认是否自动抢焦点，以及抢焦点失败时的降级策略
2. 延后 5 分钟后，是否只弹一次，还是可连续多次延后
3. 主界面打开时是全屏、居中大窗，还是普通窗口最大化
4. 悬浮窗默认形态更偏“圆形悬浮球”还是“细长条计时窗”
5. 是否允许在休息阶段直接点击“开始下一轮”

## 9. 我的想法与建议

### 9.1 产品层面

1. 最合适的主交互模型就是“悬浮窗常驻 + 主界面展开 + 到点全屏提醒”
2. 不要把第一版做成任务管理器或习惯追踪器，否则会偏离轻量工具定位
3. “全屏提醒”建议是强提醒而不是强控制，允许跳过和延后更符合真实办公环境

### 9.2 交互层面

1. 悬浮窗建议优先做“细长条”而不是纯圆球，因为更容易展示时间和状态
2. 主界面不必真的全屏，可以做成视觉上沉浸的大窗，减少系统冲突
3. 全屏休息页应突出倒计时和一两个清晰动作，不要堆过多按钮

### 9.3 技术层面

如果项目按 Python 路线推进，我建议优先使用 `PySide6`：

1. 更适合桌面应用原生能力
2. 托盘、无边框窗体、置顶、多窗口控制更稳定
3. 资源占用和启动速度更容易控制
4. 后续迁移到 Linux 也更顺

相比之下，不建议第一版用 Electron 这类更重的方案。

## 10. 实现风险

1. Windows 下全屏提醒可能与其他全屏应用、焦点管理冲突
2. 多显示器覆盖在不同分辨率和缩放比下需要额外测试
3. 悬浮窗过小会影响拖动和点击体验
4. 计时逻辑不能仅依赖简单阻塞式等待，否则窗口卡顿时会导致时间漂移

## 11. 建议的 MVP 范围

第一阶段建议只做以下内容：

1. 可拖动、置顶的小悬浮倒计时窗
2. 25 分钟专注 + 4 分钟短休息 + 4 轮后 8 分钟长休息
3. 专注结束后全屏休息提醒
4. 跳过 / 延后 5 分钟 / 关闭提醒
5. 点击悬浮窗打开主界面
6. 托盘驻留
7. 开机自启
8. 基础设置持久化

## 12. 一句话结论

这是一个很适合先做 Windows MVP 的桌面工具项目。正确方向不是做“大而全”，而是把“轻量常驻、强提醒、好看、流畅”这四件事做到位。
